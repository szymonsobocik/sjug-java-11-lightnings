JEP 320 - Remove Java EE and CORBA Modules

  • Deprecated in JDK 9

Java EE modules

  • JAX-WS (Java API for XML-Based Web Services)
  • JAXB (Java Architecture for XML Binding)
  • JAF (the JavaBeans Activation Framework)
  • Common Annotations

Na początku wersje w Java SE były identyczne z wersjami w Java EE.
Później wersje w Java EE ewoluowały:
- ficzery nie mające znaczenia dla Java SE
- problemy z utrzymaniem
- było możliwe pobranie wersji standalone i nadpisanie, ale żadko to było stosowane

CORBA

  • Dates back to 1998 (Java 1.4)
  • Evolves outside Java Community Process
  • No modern applications with CORBA in Java
dates back to 1998
- żyje osobno

Subsets of Java EE modules

  • JTA (Java Transaction API) - Java 1.3
  • J2EE Activity Service for Extended Transactions - Java 1.5

Modules removed

java.xml.ws (JAX-WS, plus the related technologies SAAJ and Web Services Metadata)
java.xml.bind (JAXB)
java.activation (JAF)
java.xml.ws.annotation (Common Annotations)
java.corba (CORBA)
java.transaction (JTA)
java.se.ee (Aggregator module for the six modules above)
jdk.xml.ws (Tools for JAX-WS)
jdk.xml.bind (Tools for JAXB)

JEP 335 - Deprecate the Nashorn JavaScript Engine

Two JDK modules terminally deprecated with
@Deprecated(forRemoval = true)

jdk.scripting.nashorn - contains the jdk.nashorn.api.scripting and jdk.nashorn.api.tree packages.

jdk.scripting.nashorn.shell - contains the jjs tool.

Will be removed with separate JEP

JEP - JDK Enhancement Proposal
https://openjdk.java.net/jeps/335

JEP 336 - Deprecate the Pack200 Tools and API

Introduced in Java SE 5.0.

Used for compressing JDK in time of 56k modems.
Later used for packaging applets for browsers.

JDK 8 is the last JDK compressed with Pack200

Historycznie było wprowadzone ze względu na wolne modemy 56k.
Służyło do pakowania JDK i aplikacji.
https://openjdk.java.net/jeps/336

Types terminally deprecated from java.base module with @Deprecated(forRemoval = true)

java.util.jar.Pack200 java.util.jar.Pack200.Packer java.util.jar.Pack200.Unpacker

API removals

Thread.destroy() - Throws NoSuchMethodError since inception (never implemented ;-))

Thread.stop(Throwable) - Throws UnsupportedOperationException since JDK 8

java.lang.Runtime.runFinalizersOnExit(boolean) (9) java.lang.SecurityManager.checkAwtEventQueueAccess() (9) java.lang.SecurityManager.checkMemberAccess(java.lang.Class,int) (9) java.lang.SecurityManager.checkSystemClipboardAccess() (9) java.lang.SecurityManager.checkTopLevelWindow(java.lang.Object) (9) java.lang.System.runFinalizersOnExit(boolean) (9)

Release Cadence

Before

Plan:

  • release by flagship features
  • major release every 2 years

Reality:

  • Java 7 took 5 years
  • Java 8 took 3 years
  • Java 9 took 3.5 years

Now

  • Fixed six-month release cadence
    March and September
  • Ship everything that is ready
  • All are major releases - no `beta versions`
Wszystko co będzie skończone będzie wypuszczane.
+ completed features get out earlier
+ no pressure to complete features on time
+ easier to invubate features
+ easier to react to changes in the ecosystem

Keep calm.

Features developped with the same pace.
Delivery rate increases.

Oracle JDK vs OpenJDK

Oracle JDK used to...

  • contain more features
  • be perceived as more stable
  • be perceived as more performant

As of Java 11, on a technical basis,
Oracle JDK and OpenJDK are identical. *

The difference is in licence and support model.

Oracle JDK is fully commercial:
from 11 on, no free use in production

Od pierwszego dnia jest płatne.
Oracle JDK license says:
"You may not:
use the Programs for any data processing or any commercial, production, or internal business purposes other than developing, testing, prototyping, and demonstrating your Application;"

New default - OpenJDK

OpenJDK is shipped by Oracle at jdk.java.net

Licenced under GPL+CE

Za to nie płacimy
OpenJDK is the new normal
GNU General Public License, version 2, with the Classpath Exception

Support

Oracle Support

Free updates for current OpenJDK version
rate: six months

Commercial support for Oracle JDK
for 5+ years for Java 11, 17, 23...
requires Oracle Java SE Subscription

No free LTS from Oracle

Shocking there is no free Long Time Support from Oracle

Fixes

1. development on "master" OpenJDK
2. merged to current version "branch" OpenJDK
3. merged to current LTS version Oracle JDK

"LTS" support means merging fixes into old JDK versions.
